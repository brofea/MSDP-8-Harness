## 三、报告内容要求

### 3.1 实验目的与系统定位

报告应说明本实验为什么位于"Agent 能调用工具"和"Agent 能安全调用工具"之间，并解释以下关系：

- Skill 负责说明能力（软约束）。
- Agent 根据用户意图生成工具请求。
- Harness 在工具执行前做代码级校验（硬约束）。
- 内容扫描在写入前检测敏感信息和注入指令。
- Tool 只执行被放行的原子操作。
- 审计日志与失败反馈用于复盘和修正。

### 3.2 Skill、Tool 与 Harness 职责边界

报告应说明：

- Skill 为什么是能力说明，而不是权限系统。
- Tool 为什么应保持原子化，不直接处理 Agent 请求。
- Harness 为什么必须作为统一执行入口。
- Prompt 软约束与 Harness 硬约束有什么区别。
- Skill 文档中的描述与 Harness 的实际策略不一致时，以谁为准。

### 3.3 工具请求协议与威胁模型

报告应解释 `tool`、`path`、`content`（文件域）、`url`（网络域）、`cmd`（命令域）等字段的含义，并列出本实验中的主要风险类型：

**文件系统域**：
- 路径穿越、未授权工具、危险文件类型、参数类型错误、写入内容过长

**网络请求域**（进阶层）：
- SSRF 攻击（域名未授权）、协议绕过（`file://` 协议）、响应过大

**Shell 命令域**（进阶层）：
- 未授权命令、管道注入、命令替换

**内容安全域**（进阶层）：
- 敏感信息泄露（API Key、手机号、身份证号）、间接提示注入、伪造对话角色

报告应说明 Harness 为什么应采用默认拒绝（fail closed）的策略。

### 3.4 实现说明

报告应结合代码说明以下文件的职责：

- `policy.py`：集中定义工作区、日志路径、工具白名单、扩展名白名单、写入长度限制、内容安全策略。
- `tools.py`：实现 `read_text`、`write_text`、`append_text`、`list_files` 等原子工具。
- `harness.py`：实现 `resolve_workspace_path`、`validate_request`、`run_tool`、`audit`、`build_feedback`。
- `main.py`：模拟 Agent 请求，观察成功、拦截和反馈。
- `test_harness.py`：用自动化测试验证护栏是否有效。

**进阶层追加说明**：
- `content_scanner.py`：实现 `scan_content` 和 `redact_content`，解释敏感信息模式和注入检测模式的设计思路。
- `network_tools.py`：实现 `validate_url` 和 `fetch_url`，解释域名白名单和协议限制。
- `command_tools.py`：实现 `validate_command` 和 `run_command`，解释命令白名单和危险操作符拦截。
- `run_real_model.py`：说明如何将真实模型输出接入 Harness 链路，解释 `parse_tool_json` 的容错处理。

### 3.5 测试与结果分析

报告应展示并解释：

- 合法读取或写入请求为什么能成功。
- 至少 4 类失败请求为什么被拦截（基础层）。
- `pytest -q` 的运行结果。
- `audit.jsonl` 中成功与失败记录的字段含义。
- `build_feedback` 如何把结构化结果转换为可回注给 Agent 的反馈。

**进阶层追加分析**：
- 内容扫描发现了哪些类型的风险（敏感信息、注入指令各至少一条）。
- 网络请求白名单如何阻止 SSRF 和协议绕过。
- Shell 命令白名单如何阻止危险操作符注入。
- 真实模型在哪些任务上产生了不合规请求，Harness 的拦截效果如何。
- 如果有多个模型，对比不同模型在工具调用参数上的稳定性差异。

## 五、注意事项

1. 不要在实验中读取真实隐私文件，`blocked/secret.txt` 只用于模拟越权测试。
2. 不要开放真实删除、格式化、系统命令执行等危险能力。
3. Shell 命令 Harness 只允许 `echo`、`date`、`wc`；不要在课堂外开放更宽的命令白名单。
4. 网络请求 Harness 只允许访问 `jsonplaceholder.typicode.com` 等公开测试域名。
5. 日志示例应使用模拟数据，不要包含真实 Token、密码、Cookie 或个人隐私信息。
6. 报告应说明每类拦截的原因，而不仅展示截图或终端输出。
7. 若使用真实大模型扩展实验，模型输出必须先转换为工具请求，再交给 Harness 校验，不能直接执行。
8. 若同学对 Skill 概念不熟悉，可先回顾实验5的 Skill 工程实战内容。

## 六、常见问题

### 基础层

#### Q1: 没有使用真实大模型会影响成绩吗？

不会。本实验基础层的重点是文件系统 Harness 的工程实现、失败反馈、审计日志和自动化安全测试，模拟 Agent 请求即可完成核心目标。内容扫描和真实模型验证属于进阶层。

#### Q2: pytest 不是必须的吗？

可以用普通 `assert` 或终端输出验证，但自动化测试更能体现工程能力。评分中"自动化安全测试"要求建议使用 `pytest -q` 展示结果。

#### Q3: 为什么要提交 `safe_file_skill/SKILL.md`？

因为本实验要求同学理解"能力说明"和"执行边界"的区别。`SKILL.md` 用于描述能力范围和禁止行为，Harness 才负责真正拦截不合规请求。此外，Skill 文档必须明确说明"与 Harness 的关系"。

#### Q4: `logs/audit.jsonl` 需要提交完整文件吗？

不要求提交很长的完整日志。报告中展示或附上示例内容即可，但示例应至少包含一次成功请求和一次失败请求。

#### Q5: 如果测试失败怎么办？

先检查是否在 `harness_guardrail_lab` 目录下运行测试，再确认 `workspace/note.md` 是否存在。若路径和文件都正确，再检查 `validate_request`、`resolve_workspace_path` 和工具白名单逻辑。

### 进阶层

#### Q6: 内容扫描的正则表达式能覆盖所有情况吗？

不能。本实验使用简化正则仅用于教学演示。真实系统中应使用更完善的内容安全方案。报告中可以讨论当前正则的局限性，这本身就是有价值的反思。

#### Q7: 网络请求的测试域名需要联网吗？

需要。`jsonplaceholder.typicode.com` 是一个公开的假数据 API，不涉及真实业务。如果课堂网络不可用，可以在报告中说明，并用模拟数据验证 Harness 校验逻辑（不实际发送网络请求）。

#### Q8: Shell 命令为什么使用 `subprocess.run(..., shell=False)`？

命令 Harness 会先用 `shlex.split` 解析命令，再检查命令白名单和危险操作符。通过校验后，使用 `shell=False` 传入参数列表执行，避免把未可信字符串交给 shell 二次解释。注意：不要在生产环境中对未校验的用户输入使用 `shell=True`。

#### Q9: `run_real_model.py` 需要什么模型？

本实验不要求特定的模型版本。实验2中部署的任意 Ollama Qwen 模型均可适用。如果默认模型不存在或响应较慢，可以用 `OLLAMA_MODEL=<模型名>` 和 `OLLAMA_TIMEOUT=<秒数>` 指定本机已有模型与超时时间。如果本机没有模型，可以跳过模块十或使用同学的模型环境验证。跳过的同学应在实验报告中说明原因。

#### Q10: 模型输出无法解析出合法 JSON 怎么办？

这正是本实验要让同学观察的关键现象——模型输出不可控。报告中应记录"模型返回了非 JSON 格式的文本，这证明了 Harness 的必要性：不能假设模型输出一定是合法的工具请求 JSON"。
